## Major

### M1. `SendQuitForDisconnectAsync` is still rate-limited; the PING/PONG exemption doesn't cover shutdown QUIT

**Severity**: Major.
**Symptom**: After commit `1ae937c` (PING/PONG exemption), QUIT on disconnect still travels through the rate-limited path. When the user-data send bucket is saturated (the exact scenario the exemption was added for), QUIT is delayed up to `SendTimeoutCancelledMs` (default 1000 ms) and may be silently dropped (the surrounding `SendQuitForDisconnectAsync` swallows `TimeoutException`). The server then sees a TCP close with no QUIT message, and channels show the user dropping with the server-default reason ("Connection reset" / "Read error"), instead of the user-supplied reason.

**Cause**:
- [src/IRCDotNet.Core/IrcClient.cs#L1605-L1620](src/IRCDotNet.Core/IrcClient.cs#L1605-L1620) — `SendQuitForDisconnectAsync` calls `SendRawWithCancellationAsync(quitMessage, ...)`.
- [src/IRCDotNet.Core/IrcClient.cs#L686-L695](src/IRCDotNet.Core/IrcClient.cs#L686-L695) — `SendRawWithCancellationAsync` rate-limits unconditionally:
  ```csharp
  if (_options.EnableRateLimit)
  {
      var rateLimitConfig = _options.RateLimitConfig ?? IrcRateLimiter.DefaultConfig;
      if (!_rateLimiter.IsAllowed("send", rateLimitConfig))
      {
          await _rateLimiter.WaitForAllowedAsync("send", rateLimitConfig, combinedCts.Token).ConfigureAwait(false);
      }
  }
  ```
  No `applyRateLimit:false` overload exists for `SendRawWithCancellationAsync`, and there is no `_isRegistered` gate (contrast with `SendRawCoreAsync` at [#L548-L553](src/IRCDotNet.Core/IrcClient.cs#L548-L553)).

**Trigger**: User submits a long bulk PRIVMSG run that exhausts the send bucket, then calls `DisconnectAsync(reason)` (or app shutdown). With default `SendTimeoutCancelledMs = 1000`, the QUIT may not reach the wire if the bucket needs >1 s to refill.

**Fix sketch**: Add an overload (or internal helper) `SendRawWithCancellationCoreAsync(message, ct, applyRateLimit)` analogous to `SendRawCoreAsync`; route `SendQuitForDisconnectAsync` through it with `applyRateLimit:false`. A graceful QUIT during shutdown is protocol-critical liveness traffic in the same sense as PING/PONG and the same rationale applies.

### M2. NICK 433 retry is unbounded and can flood the server

**Severity**: Major.
**Symptom**: `HandleNicknameInUseAsync` always falls through to a millisecond-suffixed fallback NICK after exhausting `AlternativeNicks`, but does not cap retry count and does not back off. If the network rejects every fallback (e.g., aggressive bad-nick filters, q-line, or `ERR_ERRONEUSNICKNAME`), the client keeps generating fresh NICKs and the server eventually flood-disconnects us mid-registration. The fix added in `78ded36`/`c3fe41d` resolved the original "stuck in registration with no alternatives" bug, but moved the failure mode from *silent stall* to *retry storm*.

**Cause**: [src/IRCDotNet.Core/IrcClient.cs#L2125-L2160](src/IRCDotNet.Core/IrcClient.cs#L2125-L2160).

```csharp
// Millisecond-precision suffix (4 decimal digits)...
var suffix = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds() % 10000;
_currentNick = $"{_options.Nick}{suffix:D4}";
_logger?.LogInformation("Sent NICK retry '{Nick}' after 433 with no usable alternatives", _currentNick);
await SendRawAsync($"NICK {_currentNick}").ConfigureAwait(false);
```

There is no counter, no backoff, and the same code path is re-entered every time a fresh 433 arrives.

**Trigger**: Any of: (a) network with restrictive nick policy that rejects the suffixed pattern; (b) server returning 432 instead of 433 for the suffixed nick (e.g., `_options.Nick + 4 digits` exceeds the server's `NICKLEN`); (c) genuine collision storms during netjoin/netsplit recovery.

**Fix sketch**:
1. Track `_nickRetryCount`; cap at e.g. 5 attempts; raise `ErrorReplyReceived` (or a new `RegistrationFailedEvent`) and `DisconnectInternalAsync` after exhaustion.
2. Validate the suffixed nick against `_isupportParser.MaxNicknameLength` and the RFC 2812 §2.3.1 nickname grammar before sending.
3. (Optional) introduce a small back-off (`Task.Delay(100..500ms)`) between retries.

### M3. `HandleMonitorOffline` doesn't check `_monitoredNicks` membership; can fire phantom `UserQuit` on `MONITOR -` ack

**Severity**: Major (theoretical against RFC-compliant servers; real against non-compliant ones).
**Symptom**: `HandleMonitorOffline` always creates a `_pendingMonitoredOfflines[nick]` entry and schedules a 750 ms-delayed `UserQuit`, regardless of whether the nick is still being monitored. When the application's nick-rename refresh path (`RefreshMonitoredNickAsync`) sends `MONITOR -oldnick`, a server that emits `RPL_MONOFFLINE` as an ack of removal (allowed by some implementations; the IRCv3 monitor spec is silent on this exact case) will trigger a phantom `UserQuit` for the old nick 750 ms later, even though we already raised `NickChanged(oldnick → newnick)`.

**Cause**:
- [src/IRCDotNet.Core/IrcClient.cs#L2891-L2907](src/IRCDotNet.Core/IrcClient.cs#L2891-L2907) — handler unconditionally schedules pending offline:
  ```csharp
  foreach (var target in ParseMonitorTargets(message.Parameters[^1]))
  {
      var pending = CreatePendingMonitoredOffline(message, target.Nick);
      _pendingMonitoredOfflines[target.Nick] = pending;
      SafeFireAndForget(() => FinalizePendingMonitoredOfflineAsync(pending), $"MonitorOffline_{target.Nick}");
  }
  ```
- [src/IRCDotNet.Core/IrcClient.cs#L2998-L3010](src/IRCDotNet.Core/IrcClient.cs#L2998-L3010) — finalize fires even if oldnick is gone from `_monitoredNicks`:
  ```csharp
  if (!_pendingMonitoredOfflines.TryGetValue(pending.Nick, out var current) || current.Timestamp != pending.Timestamp)
      return;
  _pendingMonitoredOfflines.TryRemove(pending.Nick, out _);
  _userInfo.TryRemove(pending.Nick, out _);
  RaiseEventAsync(UserQuit, new UserQuitEvent(pending.Message, pending.Nick, pending.User, pending.Host));
  ```
- [src/IRCDotNet.Core/IrcClient.cs#L2985-L2995](src/IRCDotNet.Core/IrcClient.cs#L2985-L2995) — refresh path issues `MONITOR -` then `MONITOR +` without correlating server acks.

**Trigger**: A monitored user renames; `ApplyNickChange` fires `RefreshMonitoredNickAsync(old, new)`; server acks the removal with `RPL_MONOFFLINE old`; no `RPL_MONONLINE` for `old` follows; 750 ms later, phantom `UserQuit(old)`.

**Spec**: IRCv3 MONITOR (<https://ircv3.net/specs/extensions/monitor>) defines `RPL_MONONLINE`/`RPL_MONOFFLINE` for *status-changed* events; behavior for `MONITOR -` is implementation-defined. Conforming servers (e.g., charybdis/solanum) do **not** emit `MONOFFLINE` on `MONITOR -`, so this finding is theoretical against them. UnrealIRCd and InspIRCd should be verified.

**Fix sketch**: In `HandleMonitorOffline`, gate on `_monitoredNicks.Contains(target.Nick)` before creating the pending offline. Also clear `_pendingMonitoredOfflines[oldNick]` *after* the `MONITOR -` is acked, not before the send (currently only cleared at the start of `ApplyNickChange`).

### M4. `HandleNotice` silently drops echoed CTCP replies even though `NoticeEvent.IsEcho` exists

**Severity**: Major.
**Symptom**: `1db9dae` added `IsEcho` to `NoticeEvent` and propagated it from `HandleNotice` to `RaiseEventAsync(NoticeReceived, ...)` for the non-CTCP branch, but the CTCP-reply branch still has the original `if (isEchoNotice) return;` short-circuit. The application cannot observe its own outgoing CTCP replies (`VERSION`, `TIME`, `PING`, `USERINFO`, etc.) — the `IsEcho` flag in `NoticeEvent` is effectively unreachable for CTCP-reply notices. The `HandleCtcpRequest` ACTION path was correctly updated (raises `CtcpActionEvent` with `IsEcho=true`); the symmetrical NOTICE/CTCP-reply path was not.

**Cause**: [src/IRCDotNet.Core/IrcClient.cs#L1858-L1868](src/IRCDotNet.Core/IrcClient.cs#L1858-L1868):

```csharp
// CTCP reply detection: NOTICE messages starting with \x01 are CTCP replies.
if (text.Length >= 2 && text[0] == '\u0001')
{
    // Skip echoed CTCP replies — we sent this reply ourselves
    if (isEchoNotice) return;       // <- silently dropped, no event raised

    var ctcpContent = text[^1] == '\u0001' ? text[1..^1] : text[1..];
    HandleCtcpReply(message, nick, user, host, ctcpContent);
    return;
}
```

**Trigger**: User app records outgoing CTCP traffic for diagnostics or inbox display. With `echo-message` enabled, our own `VERSION` reply NOTICE comes back over the wire; we drop it.

**Fix sketch**: Either:
- (preferred) Raise `NoticeReceived` with `IsEcho=true` and the raw CTCP-wrapped text intact, OR
- Add a `CtcpReplyEvent` with an `IsEcho` flag and route echoed replies to it.

### Med1. `IsupportParser` state is not reset on disconnect/reconnect; ISUPPORT values from the previous server can leak to the next session

**Severity**: Medium.
**Symptom**: `_isupportParser` is created once in the constructor ([src/IRCDotNet.Core/IrcClient.cs#L280](src/IRCDotNet.Core/IrcClient.cs#L280)) and is never reset across reconnect/disconnect cycles. New 005 messages overwrite same-keyed features, but **features the new server does not advertise retain values from the previous server**. After this commit range, this surfaces directly through:

- the new typed `IsupportReceivedEvent` ([src/IRCDotNet.Core/Events/IrcEvents.cs#L527-L590](src/IRCDotNet.Core/Events/IrcEvents.cs#L527-L590)) — its first-005 snapshot inherits stale fields,
- the new `GetServerCaseMapping()` accessor ([src/IRCDotNet.Core/IrcClient.cs#L3487](src/IRCDotNet.Core/IrcClient.cs#L3487)),
- existing `GetServerNetworkName / GetServerMaxNicknameLength / GetServerChannelTypes`.

**Cause**: `DisconnectInternalAsync` clears `_channels`, `_userInfo`, `_enabledCapabilities`, `_supportedCapabilities`, `_monitoredNicks`, `_pendingNamesUsers`, `_pendingMonitoredOfflines`, `_motdBuffer`, etc. but does **not** call `_isupportParser.Clear()` ([src/IRCDotNet.Core/Protocol/IsupportParser.cs#L323-L327](src/IRCDotNet.Core/Protocol/IsupportParser.cs#L323-L327)).

**Trigger**: User changes server endpoint and reconnects (e.g., switches from Libera.Chat to OFTC). Old `NETWORK="Libera.Chat"`, `CASEMAPPING=rfc1459`, `CHANTYPES=#`, etc. survive until the new 005 burst overwrites them — and any field not advertised by the new server stays stale forever.

**Fix sketch**: Call `_isupportParser.Clear()` in `DisconnectInternalAsync` (or at the top of `ConnectAsync`) before the new 005 burst arrives. Note that `IsupportParser.Clear()` already exists; it just isn't wired up.

### Med2. `RefreshMonitoredNickAsync` partial failure leaves `_monitoredNicks` and the server's monitor list out of sync

**Severity**: Medium.
**Symptom**: When a monitored nick renames, `ApplyNickChange` updates `_monitoredNicks` (`Remove(oldNick)`, `Add(newNick)`) **before** the server-side refresh starts ([src/IRCDotNet.Core/IrcClient.cs#L2948-L2950](src/IRCDotNet.Core/IrcClient.cs#L2948-L2950)). `RefreshMonitoredNickAsync` then sends `MONITOR -oldNick` and `MONITOR +newNick` sequentially. If the second send fails (transient transport error, IOException, server-side parse error), `_monitoredNicks` locally claims to monitor `newNick` but the server only has the removal applied — `newNick` is not actually being monitored.

**Cause**: [src/IRCDotNet.Core/IrcClient.cs#L3014-L3030](src/IRCDotNet.Core/IrcClient.cs#L3014-L3030):

```csharp
private async Task RefreshMonitoredNickAsync(string oldNick, string newNick)
{
    if (!SupportsMonitorCapability() || !_isConnected) return;
    try
    {
        await SendRawAsync($"MONITOR - {oldNick}").ConfigureAwait(false);
        await SendRawAsync($"MONITOR + {newNick}").ConfigureAwait(false);
    }
    catch (Exception ex)
    {
        _logger?.LogDebug(ex, "Failed to refresh MONITOR target from {OldNick} to {NewNick}", oldNick, newNick);
    }
}
```

State is never reverted on partial failure.

**Fix sketch**: Either revert `_monitoredNicks` mutations on exception, or atomically combine into a single `MONITOR -oldNick,newNick` then `MONITOR +newNick` (ircv3 MONITOR allows comma-separated nick lists), or queue a re-sync of the monitor list to the server on next connect (track desired vs. confirmed monitored sets).

### Med3. `DisposeAsync` does not actually wait for queued events to drain before returning

**Severity**: Medium.
**Symptom**: Commit `879ec30`'s message says: *"Keep queued event dispatches alive until already-enqueued handlers finish during DisposeAsync, then close further dispatch and finalize disconnect state."* The implementation lets queued dispatches **continue running** on a fire-and-forget task, but does not synchronously **wait** for the dispatch loop to finish before `DisposeAsync` returns. Subscribers can therefore fire event handlers *after* the user's `await client.DisposeAsync()` has returned, after the user has begun cleanup of their own subscribers. In practice this can surface as `ObjectDisposedException` inside subscribers that race with user-side cleanup; those exceptions are swallowed inside `RaiseEventAsync`'s per-handler try/catch and logged, so behavior is bounded but the contract suggested by the commit message is stronger than what the code actually delivers.

**Cause**: [src/IRCDotNet.Core/IrcClient.cs#L2572-L2592](src/IRCDotNet.Core/IrcClient.cs#L2572-L2592):

```csharp
public async ValueTask DisposeAsync()
{
    if (Interlocked.Exchange(ref _disposeRequested, 1) == 1) return;
    try
    {
        await DisconnectInternalAsync("Client disposed").ConfigureAwait(false);
    }
    catch (Exception ex) { ... }
    finally
    {
        Interlocked.Exchange(ref _eventDispatchClosed, 1);   // blocks new enqueues only
        Interlocked.Exchange(ref _disposed, 1);
        _sendLock?.Dispose();
        _connectLock?.Dispose();
        GC.SuppressFinalize(this);
    }
}
```

There is no `await` on the dispatch loop's completion. The pinned test [tests/IRCDotNetCore.Tests/IrcClientTests.cs#L505-L555](tests/IRCDotNetCore.Tests/IrcClientTests.cs#L505-L555) (`DisposeAsync_WhenEventsAreAlreadyQueued_ShouldDrainQueueAndRaiseDisconnected`) only asserts that the events fire **eventually within 5 s**, not that they fire before `disposeTask` completes — so the test passes even though the contract is fuzzy.

**Fix sketch**: Add a `TaskCompletionSource` set when `ProcessPendingEventDispatchesAsync` exits with the queue empty; in `DisposeAsync`'s finally, after setting `_eventDispatchClosed = 1`, await that TCS (with a generous bound, e.g. `Task.WhenAny(drainTcs.Task, Task.Delay(_options.SendTimeoutCancelledMs * 2))`). Then update the test to assert ordering (`disposeTask` completes only after `secondHandlerCompleted` and `disconnectedReceived`).

### Med4. NAMES race: another user's JOIN that arrives before the first 353 can be lost from the channel roster after `RPL_ENDOFNAMES`

**Severity**: Medium.
**Symptom**: When we ourselves join a channel, the server typically sends `JOIN (us) → 353 (NAMES) → 366 (ENDOFNAMES)`. If another user's JOIN message for the same channel arrives between our self-JOIN and the first 353 (possible on bouncer playback, federated IRCds, or netsplit recovery — the IRC protocol does **not** mandate strict ordering between member JOIN notifications and our NAMES reply on a brand-new channel), the new buffering machinery introduced by `afad2ff` silently drops their JOIN: the live `_channels[channel].Add(nick)` is overwritten when 366 fires the snapshot+mutations replacement, and `QueuePendingNamesAdd` is a no-op because `_pendingNamesUsers[channel]` doesn't exist yet.

**Cause**:
- [src/IRCDotNet.Core/IrcClient.cs#L1745-L1752](src/IRCDotNet.Core/IrcClient.cs#L1745-L1752) — other-user JOIN on a not-yet-NAMES'd channel:
  ```csharp
  else
  {
      if (_channels.ContainsKey(channel))
      {
          _channels[channel].Add(nick);          // live add (transient)
          InvalidateChannelsCache();
      }
      QueuePendingNamesAdd(channel, nick);       // no-op until first 353
  }
  ```
- [src/IRCDotNet.Core/IrcClient.cs#L2059-L2066](src/IRCDotNet.Core/IrcClient.cs#L2059-L2066):
  ```csharp
  private void QueuePendingNamesAdd(string channel, string nick)
  {
      if (_pendingNamesUsers.TryGetValue(channel, out var pendingNamesState))
          pendingNamesState.Mutations.Enqueue(new PendingNamesMutation(PendingNamesMutationKind.Add, nick));
  }
  ```
- [src/IRCDotNet.Core/IrcClient.cs#L2028-L2050](src/IRCDotNet.Core/IrcClient.cs#L2028-L2050) — 366 path replaces `_channels[channel]` outright with `replacementSet` built from the snapshot+mutations.

**Trigger**: Bouncer (ZNC) reconnect playback that interleaves member JOINs with the buffered NAMES reply; a multi-server IRCD where another server's user join propagates over the link before the local server's 353 is sent; manual test scenario where a user `/JOIN`s simultaneously.

**Fix sketch**:
- Pre-allocate `_pendingNamesUsers[channel] = new PendingNamesState()` at self-JOIN ([src/IRCDotNet.Core/IrcClient.cs#L1738](src/IRCDotNet.Core/IrcClient.cs#L1738)) so the mutation queue exists from the moment we expect a NAMES burst, *or*
- In `HandleEndOfNames`, merge the existing `_channels[channel]` live set into `usersByNick` before applying mutations — i.e., union of (snapshot ∪ live set ∪ replayed mutations) rather than replace.

The same gap applies symmetrically to QUIT, KICK, NICK, PART arriving before the first 353. Tests should cover at least one of those cases too.

### Med5. `HandleNotice`'s NickServ auto-IDENTIFY heuristic is not gated on `isEchoNotice`

**Severity**: Medium (small attack surface, theoretical edge case).
**Symptom**: For non-CTCP notices, `HandleNotice` propagates `isEchoNotice` correctly to `NoticeReceived`, but the NickServ auto-IDENTIFY block earlier in the same method is *not* gated on `!isEchoNotice`. With `echo-message` enabled, if our own outgoing NOTICE happens to match the trigger condition (`source nick == "NickServ"` AND text contains "IDENTIFY"), the client could fire `IDENTIFY` against itself. Reachable only when the user's own nick is `NickServ`, which is rare but not impossible (services impersonation, test fixtures, registered services-team operator).

**Cause**: [src/IRCDotNet.Core/IrcClient.cs#L1869-L1885](src/IRCDotNet.Core/IrcClient.cs#L1869-L1885):

```csharp
{

    SafeFireAndForget(async () =>
    {
        await SendMessageAsync("NickServ", $"IDENTIFY {_currentNick} {_options.NickServPassword}").ConfigureAwait(false);

    }, "NickServIdentify");

```

**Fix sketch**: Add `!isEchoNotice` to the conditional so echoed self-notices can never trigger auto-IDENTIFY. Trivial one-line change; raises symmetry with the CTCP-reply echo guard a few lines up.

### Min1. `PrivateMessageEvent.IsChannelMessage`, `CtcpActionEvent.IsChannelAction`, and `TypingIndicatorEvent.IsChannelTyping` ignore ISUPPORT CHANTYPES; they hardcode `#&+!`

**Severity**: Minor.
**Symptom**: Commit `1db9dae`/`89908c3` correctly extended the channel-prefix list from `{#, &}` to `{#, &, +, !}` matching the RFC 2812 §1.3 default CHANTYPES, but the events do not consult the server's actual `_isupportParser.ChannelTypes`. Networks that advertise a non-default `CHANTYPES` (e.g., `#` only on most modern IRCds, `~` on some legacy networks) will misclassify channel-vs-PM at the event level. The `_channels` dictionary uses ordinal-case-insensitive keys and works correctly regardless; the bug is contained to event-level classification.

**Cause**:
- [src/IRCDotNet.Core/Events/IrcEvents.cs#L161](src/IRCDotNet.Core/Events/IrcEvents.cs#L161) — `IsChannelMessage` hardcoded.
- [src/IRCDotNet.Core/Events/IrcEvents.cs#L1066](src/IRCDotNet.Core/Events/IrcEvents.cs#L1066) — `IsChannelAction` hardcoded.
- [src/IRCDotNet.Core/Events/IrcEvents.cs#L1235-L1239](src/IRCDotNet.Core/Events/IrcEvents.cs#L1235-L1239) — `IsChannelTyping` hardcoded (this is the one that has the defensive `Target.Length > 0` check; the other two rely on `string.StartsWith(char)` returning false for empty strings, which is also safe).

**Spec**: RFC 2812 §1.3 default CHANTYPES = `#&+!`; ISUPPORT CHANTYPES (modern.ircdocs.horse) overrides per-network.

**Fix sketch**: Pass the parsed CHANTYPES into events at construction time (preferred), or switch `IsChannelMessage`/`IsChannelAction`/`IsChannelTyping` from instance properties to a method that takes the client (`bool IsChannel(IIrcClient client)`). Either way the events become self-contained.

### Min2. `MonitorOfflineCorrelationWindow` is hardcoded at 750 ms

**Severity**: Minor.
**Symptom**: [src/IRCDotNet.Core/IrcClient.cs#L21](src/IRCDotNet.Core/IrcClient.cs#L21) fixes the explicit-rename correlation window at 750 ms with no `IrcClientOptions` knob. On very high-latency connections (mobile / LTE backhaul / cross-continent VPN), an explicit `NICK` rename signal can legitimately arrive after 750 ms, and `FinalizePendingMonitoredOfflineAsync` will then fire a real `UserQuit(oldNick)` instead of the rename being correlated. **Cause**: [src/IRCDotNet.Core/IrcClient.cs#L21](src/IRCDotNet.Core/IrcClient.cs#L21).

**Fix sketch**: Add `MonitorOfflineCorrelationWindowMs` (default 750) to `IrcClientOptions`, validate `> 0`, use it in `FinalizePendingMonitoredOfflineAsync`.

### Min3. `ApplyNickChange` overwrites pre-existing `_userInfo[newNick]` entries

**Severity**: Minor.
**Symptom**: On rename, [src/IRCDotNet.Core/IrcClient.cs#L2932-L2941](src/IRCDotNet.Core/IrcClient.cs#L2932-L2941) does:

```csharp
if (_userInfo.TryRemove(oldNick, out var userInfo))
{
    userInfo.Nick = newNick;
    if (!string.IsNullOrEmpty(user) && string.IsNullOrEmpty(userInfo.User)) userInfo.User = user;
    if (!string.IsNullOrEmpty(host) && string.IsNullOrEmpty(userInfo.Host)) userInfo.Host = host;
    _userInfo[newNick] = userInfo;       // overwrites if newNick already had data
}
```

If `_userInfo` already had an entry for `newNick` (e.g., a prior user with that nick whose `UserQuit` we missed, or stale data from a 436 collision recovery), the existing entry is silently overwritten by `userInfo` from `oldNick`. Because the merge logic only fills empty fields, partial data loss is unlikely but possible (e.g., the existing entry's `Account` or `RealName` is dropped if `userInfo` doesn't carry them).

**Fix sketch**: Either `_userInfo.AddOrUpdate(newNick, _ => userInfo, (_, existing) => MergePreferringNonEmpty(existing, userInfo))`, or document the precedence policy explicitly and accept the overwrite.

### Min4. `HandleNickChange` `TrimStart(':')` is over-aggressive

**Severity**: Minor.
**Symptom**: [src/IRCDotNet.Core/IrcClient.cs#L1894](src/IRCDotNet.Core/IrcClient.cs#L1894):

```csharp
var newNick = message.Parameters[0].TrimStart(':');
```

`IrcMessage.Parse` already strips the trailing-parameter `:` prefix, so this `TrimStart` should be a no-op for well-formed messages. For malformed `::foo` it strips both colons, which would silently mask a server bug. Per RFC 2812 §2.3.1, `:` is not a valid nickname character, so the trimmed-down `foo` is "more correct" than `::foo` — but this defensive trim hides the malformation rather than logging it.

**Fix sketch**: Remove the `TrimStart`; if a server is sending `::foo` it should fail noisily (or be `WarnOrThrow`-routed via the existing diagnostic logging path). Pre-existing code paths use the raw `message.Parameters[0]` directly without `TrimStart`.

---

### Min5. NICK fallback doesn't validate the resulting nick against `_isupportParser.MaxNicknameLength` or RFC 2812 §2.3.1 grammar

**Severity**: Minor.
**Symptom**: [src/IRCDotNet.Core/IrcClient.cs#L2154-L2160](src/IRCDotNet.Core/IrcClient.cs#L2154-L2160) constructs `$"{_options.Nick}{suffix:D4}"` without checking length against the network's `NICKLEN`. If `_options.Nick.Length + 4 > MaxNicknameLength`, the server returns 432 `ERR_ERRONEUSNICKNAME` (or trims silently), and the client may either retry into M2's flood, or end up with a nick that doesn't match the server's view.

Same concern applies to the `AlternativeNicks` path at [#L2143](src/IRCDotNet.Core/IrcClient.cs#L2143) (pre-existing) — but the `78ded36`/`c3fe41d` commits made the fallback path the new default for the no-alternatives case, expanding the surface area.

**Fix sketch**: Truncate `_options.Nick` to `MaxNicknameLength - 4` before appending the suffix; validate the grammar regex (`[A-Za-z\[\]\\\`_^{|}][A-Za-z0-9\[\]\\\`_^{|}-]{0,8}` per RFC 2812) and fall back to a known-safe template (e.g. `Guest{suffix}`) if the user-supplied base is non-conforming.

## Notes

### N1. `SendRawWithCancellationAsync` rate-limit gating is inconsistent with `SendRawCoreAsync`

`SendRawCoreAsync` at [#L548-L553](src/IRCDotNet.Core/IrcClient.cs#L548-L553) gates on `applyRateLimit && _options.EnableRateLimit && _isRegistered`; `SendRawWithCancellationAsync` at [#L686-L695](src/IRCDotNet.Core/IrcClient.cs#L686-L695) gates only on `_options.EnableRateLimit`. So rate-limited sends from `SendMessageWithCancellationAsync` and `SendQuitForDisconnectAsync` are subject to throttling even pre-registration, where the same call via the non-cancellation overload would not be. Pre-registration sends rarely happen via `SendMessageWithCancellationAsync` so the practical impact is low, but the asymmetry should be eliminated when M1 is fixed.

### N2. `Dispose()` (sync) drops queued events without draining

The synchronous `Dispose()` at [#L2526-L2570](src/IRCDotNet.Core/IrcClient.cs#L2526-L2570) sets `_eventDispatchClosed = 1` and `_disposed = 1` immediately, then disposes locks. Pending events in `_pendingEventDispatches` continue to run on the fire-and-forget dispatcher, but the caller has no way to wait for them. This is acceptable as a documented limitation (sync `IDisposable` cannot wait without blocking the thread), but the public XML doc should call this out and recommend `IAsyncDisposable.DisposeAsync` for graceful shutdown.

### N3. Server-time tag parsing is permissive

[src/IRCDotNet.Core/Events/IrcEvents.cs#L33-L43](src/IRCDotNet.Core/Events/IrcEvents.cs#L33-L43) uses `DateTimeOffset.TryParse(timeValue, InvariantCulture, AssumeUniversal | AdjustToUniversal, ...)`. This accepts more formats than the IRCv3 server-time spec (<https://ircv3.net/specs/extensions/server-time>) requires (which is RFC 3339, e.g. `2011-10-19T16:40:51.620Z`). Permissive parsing is robust for production but means malformed tags silently fall back to `DateTimeOffset.UtcNow` instead of being logged. `DateTimeOffset.TryParseExact` against the canonical format with a logged fallback would be more diagnostic-friendly.

Also note the **behavior change**: pre-`e848a0b`, `IrcEvent.Timestamp` was always local-receive time; now it is server-time when the tag is present. Consumers that rely on `Timestamp` for monotonic ordering may see non-monotonic sequences during bouncer playback / `CHATHISTORY` replay (every replayed message carries its original server-time). The CHANGELOG and README should call this out as a behavior change for downstream callers (the IRCDotNet.Client app uses a separate monotonic event sequence and is not affected).

### N4. `SupportsMonitorCapability` allows MONITOR commands when only `extended-monitor` is enabled

[src/IRCDotNet.Core/IrcClient.cs#L2865-L2869](src/IRCDotNet.Core/IrcClient.cs#L2865-L2869):

```csharp
private bool SupportsMonitorCapability()
{
    return _enabledCapabilities.Contains(IrcCapabilities.MONITOR)
        || _enabledCapabilities.Contains(IrcCapabilities.EXTENDED_MONITOR);
}
```

Per IRCv3 (<https://ircv3.net/specs/extensions/extended-monitor>), `extended-monitor` is a strict superset of `monitor` and requires it as a dependency. A conforming server cannot advertise `extended-monitor` without `monitor`. The `||` here is harmless against compliant servers but masks misconfiguration on non-compliant ones (e.g., an experimental ircd that advertises only `extended-monitor`). Consider `&& serverSupportsMonitor` if defensive.
